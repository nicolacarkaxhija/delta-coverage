# Functional Specifications — delta-coverage

## Purpose

`delta-coverage` enforces code coverage thresholds exclusively on files and lines that were **modified in the current Git branch**, rather than on the entire codebase. This solves a common enterprise problem: large codebases with historical technical debt where global coverage requirements block developers from merging new, well-tested code.

## Target Audience

- Engineering teams working in large, legacy codebases with pre-existing untested code
- CI/CD pipeline maintainers who want to enforce coverage on new work without a codebase-wide gate
- Developers using any test framework that outputs Istanbul/nyc-compatible JSON coverage reports

## Core Concept

Instead of asking "does the whole project have 80% coverage?", delta-coverage asks "do the lines *I changed* have 80% coverage?". Developers are held accountable only for the code they wrote.

## Supported Coverage Formats

| Format | File | Generated by |
|--------|------|-------------|
| Summary | `coverage-summary.json` | Istanbul/nyc, Vitest, Jest |
| Final (line-level) | `coverage-final.json` | Istanbul/nyc, Vitest, Jest |

Both files are typically found in the `./coverage` directory after running tests.

## Enforcement Modes

### File Mode (default)

Reads `coverage-summary.json`. For each file changed in the branch, checks whether the file's total line coverage percentage meets the threshold.

- **Simpler** to understand and debug
- A 1-line change to a legacy file with 0% coverage will fail, even if the changed line is covered
- Best for greenfield projects or codebases that are already well-tested

### Line Mode

Reads `coverage-final.json`. Uses `git diff` to identify the exact line ranges that changed. Maps those lines to statement IDs in the coverage data, then calculates:

```
covered modified statements
─────────────────────────── × 100
total modified statements
```

- **More precise**: developers are accountable only for the statements they changed
- A legacy file with 0% global coverage is fine, as long as the new lines are covered
- Ideal for large enterprise codebases with existing technical debt

## Configuration

Configuration can be provided via:
- `.deltarc` or `.deltarc.json` (JSON)
- `.deltarc.yaml` / `.deltarc.yml`
- `.deltarc.js` / `delta-coverage.config.js`
- `"delta-coverage"` key in `package.json`
- CLI flags (override config file values)

### Options

| Option | Default | Description |
|--------|---------|-------------|
| `base` | `origin/master` | Git ref to compare against (branch, tag, or commit SHA) |
| `threshold` | `80` | Minimum coverage percentage (0–100) |
| `checkMode` | `file` | `file` or `line` |
| `coverageDir` | `./coverage` | Directory containing coverage JSON files |
| `include` | — | Glob patterns to restrict which changed files are checked |
| `exclude` | — | Glob patterns to skip files from checking |
| `onMissingCoverage` | `warn` (file) / `fail` (line) | What to do when a changed file has no coverage data |
| `debug` | `false` | Print verbose path comparison logs |

### Example `.deltarc`

```json
{
  "base": "origin/main",
  "threshold": 90,
  "checkMode": "line",
  "coverageDir": "./coverage",
  "include": ["src/**/*.ts"],
  "exclude": ["src/**/*.d.ts", "src/generated/**"]
}
```

## CLI Usage

```sh
delta-coverage [options]

Options:
  --base <ref>           Git ref to diff against (default: origin/master)
  --threshold <number>   Coverage % required (default: 80)
  --check-mode <mode>    "file" or "line" (default: file)
  --coverage-dir <path>  Path to coverage directory (default: ./coverage)
  --debug                Enable verbose debug output
  -h, --help             Display help
```

### Exit Codes

| Code | Meaning |
|------|---------|
| `0` | All changed files pass the threshold, or no files were changed |
| `1` | One or more files fail the threshold, or a fatal error occurred |

## Programmatic API

```ts
import { runDeltaCheck } from 'delta-coverage';

const report = await runDeltaCheck({
  base: 'origin/main',
  threshold: 85,
  checkMode: 'line',
  coverageDir: './coverage',
});

console.log(report.passed);         // boolean
console.log(report.summary);        // { total, passed, failed, skipped, missing }
console.log(report.files);          // FileResult[]
```

### `DeltaReport`

```ts
interface DeltaReport {
  passed: boolean;
  files: FileResult[];
  summary: {
    total: number;
    passed: number;
    failed: number;
    skipped: number;
    missing: number;
  };
}

interface FileResult {
  file: string;
  status: 'passed' | 'failed' | 'skipped' | 'missing';
  pct?: number;       // calculated coverage percentage
  threshold?: number; // applied threshold
  message?: string;   // reason for skipped/missing status
}
```

## CI/CD Integration

### GitHub Actions

```yaml
- name: Run tests with coverage
  run: npm test -- --coverage

- name: Check delta coverage
  run: npx delta-coverage --base origin/main --threshold 80 --check-mode line
```

### GitLab CI

```yaml
coverage:
  script:
    - npm test -- --coverage
    - npx delta-coverage --base origin/main --threshold 80
```

## Known Limitations

1. **Requires a Git repository**: `delta-coverage` calls `git` commands directly. It must be run inside a git working tree.
2. **Coverage must exist before running**: run your test suite with coverage enabled before invoking `delta-coverage`.
3. **Binary and generated files**: files with no executable statements (e.g. type declaration files, auto-generated code) should be excluded via the `exclude` option.
4. **Squash merges**: when using squash merge workflows, the `--base` ref must point to the target branch's current HEAD, not a merge-base commit.
